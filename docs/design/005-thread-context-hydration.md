# Design 005: Thread Context Hydration

**Status:** Proposed
**Date:** 2026-07-09
**Scope:** Give a new task-run the context it needs about a returning customer
and their ongoing thread, without replaying a raw transcript. Closes the
Phase 3b gap left open by `004-domain-remodel.md` §13 (and the older `003` §8
limitation): a fresh run currently starts cold, so a long-time customer is met
as a stranger every time.

Builds on `000-foundation.md`, `003-dispatcher-as-participant.md`, and
`004-domain-remodel.md`. Changes nothing in the agentkit Action pipeline.

---

## 1. The problem

After `004`, a conversation is a **persistent thread** (one per customer ×
channel), durable across many runs and cases. A run is one intake *task*; when
a returning customer sends a new message and the previous run has finished, the
Router starts a *fresh* run bound to the same thread (`004` §6).

That fresh run's LLM context is seeded from `AgentLoopInput.Messages`, which is
empty. So the agent has no idea who this is, what was said last week, or that
there are three closed jobs on file. It re-asks for the name and address it
already knows. For a one-shot intake that was tolerable; for the omnichannel
CRM the product is becoming, it is the whole game.

**The naive fix — replay the full thread transcript into the run — is wrong**,
for three reasons:

- **Cost & window.** The context is re-sent on *every* turn of every run. A
  multi-year thread blows the context window and the bill, and grows without
  bound.
- **Signal.** The one fact that matters for *this* job drowns in years of
  chit-chat.
- **Correctness.** Replaying prior *assistant / tool* turns verbatim invites
  the agent to believe it already called those tools *in this run* — so it
  re-issues actions, or misjudges how far along the current case is. Prior turns
  are history, not this run's working memory, and must be framed as such.

So hydration is not "load the transcript." It is "assemble a compact, bounded,
clearly-labeled **briefing**."

## 2. The model: tiered memory

Give each new run a briefing built from four tiers, cheapest and most durable
first. Only the first three ship here; the fourth is the escape hatch.

1. **Customer profile (CRM facts).** Name, known service addresses, and a
   one-line-per prior case list (`type · status · when`). Small, high-signal,
   grows slowly. *This is what actually serves a long-time customer* — the value
   is the distilled facts, not the verbatim chat. Sourced from `customers` +
   `contact_identities` + `cases`, all of which exist after `004`.

2. **Rolling thread summary.** A maintained ~paragraph digest of everything
   older than the recent window ("Regular customer; 3 prior jobs, last a leaking
   water heater in March, resolved. Prefers morning visits."). Regenerated by an
   LLM activity at `close_case`. This is what bounds *unbounded* history into a
   *fixed-size* artifact — the mechanism that lets tier 3 stay small forever.

3. **Recent transcript window.** The last *N* messages on the thread (or a
   token budget, whichever is smaller), most-recent-first, verbatim, under an
   explicit `--- recent messages (context, already sent) ---` heading.
   Older messages are elided with a marker; they are represented only by
   tiers 1–2.

4. **Semantic recall (deferred).** When something relevant is buried deeper than
   the window and too specific for the summary, retrieve it: embed messages/cases
   and pull the top-k for the current message. Real cost, real value only once
   threads are long and varied. Explicitly **not built now** — noted so the
   briefing assembler has an obvious place to grow.

The briefing is assembled fresh at the **start of each new task-run** (so it
reflects the latest profile, summary, and window), then held stable for that
run's life.

## 3. Where it plugs in (this codebase)

Two facts about the current loop decide the design:

- The run's LLM context starts from `AgentLoopInput.Messages` (empty today), and
  the system prompt comes from a *static* `AgentDefinition.System` string, used
  in the completion activity.
- The scripted fake-LLM (`DISPATCH_FAKE_LLM=1`) picks its script line by
  **counting user turns**. This is exactly why `004` deferred hydration:
  injecting prior messages as user turns would desync the count.

Both point to the same answer: **carry the briefing as system context, not as
replayed user/assistant turns.**

### 3.1 The seam

Add one generic field to the agent-loop input:

```go
// AgentLoopInput
SystemContext string // per-run addendum appended to the definition's System
                     // prompt; the app fills it, agentkit only carries it.
```

The completion activity concatenates `def.System + "\n\n" + input.SystemContext`
when non-empty. That is the *entire* agentkit change, and it stays
business-agnostic: agentkit knows only "a run may carry extra system context,"
never what a briefing is. **Hard rule 1 (agentkit ⊥ app) holds.**

Because the briefing lands as *system* context and adds **zero user turns**, the
fake-LLM's turn counting is untouched — the deferred blocker dissolves rather
than needing special handling.

### 3.2 The assembler (app side)

The `Router` already resolves customer, thread, and playbook on every inbound
message, and it is app code — free to read the domain store. When it starts a
*new* run (not when it reuses a live one), it calls a new
`domain` assembler:

```go
briefing, err := store.HydrateThreadContext(ctx, orgID, conv.ID, customer.ID)
// ... CreateRun; CreateRunBinding; SignalWithStart(..., AgentLoopInput{
//       RunID, OrgID, Agent, SystemContext: briefing })
```

`HydrateThreadContext` assembles tiers 1–3 into one text block. It is a plain
store read (tiers 1 & 3) plus a stored-column read (tier 2), so it stays out of
the workflow — no determinism concern. A reused live run is *not* re-hydrated:
its briefing is already in-flight and its within-run history is authoritative.

### 3.3 Rolling summary maintenance

At `close_case`, an activity summarizes `{prior thread summary + this case's
transcript}` into the new thread summary and stores it. Home: a
`conversations.summary` column (thread-scoped, one row, naturally overwritten —
no conflict with the append-only `events` rule, which governs the event log, not
projections). This is the one added LLM call; it runs once per completed case,
off the customer's critical path.

## 4. Prompt-cache shape

The briefing is re-sent every turn of a run, so order it **stable-prefix-first**
for Anthropic prompt caching: profile and summary (rarely change within a run)
lead; the recent-window tail is the volatile part. Over a run's turns the
expensive prefix is cached and amortized; only the small tail re-tokenizes.

## 5. Phasing

- **5a — Briefing skeleton (minimal slice).** `SystemContext` seam + assembler
  producing **tier 1 (profile-lite: name + prior-case count/list) and tier 3
  (recent window)**. No summarization LLM call, no retrieval. This alone ends
  "returning customer starts cold" and is low-risk and fully reversible. Ship
  first.
- **5b — Rolling summary (tier 2).** Add the `close_case` summarizer + the
  `conversations.summary` column; fold it into the briefing ahead of the window.
  Do this when threads grow past what a raw window can represent.
- **5c — Semantic recall (tier 4).** Only when long, varied threads make buried
  facts a real miss. Its own follow-up; likely rides the same embedding
  infrastructure a search/inbox feature would need anyway.

Each slice is shippable and reversible alone.

## 6. Non-goals / open questions

- **Cross-thread / cross-channel profile.** A customer can have several threads
  (`004` §3–4). This doc hydrates *one thread's* run from that thread + the
  customer's case list. A briefing that pulls the customer's *other* active
  threads is a natural extension, but waits on the unified customer-profile UI
  (`004` §13, deferred) that would motivate it.
- **Window size / token budget** (tier 3) and **summary length** (tier 2) are
  tuning knobs, not architecture; start at a small N and a short summary and
  adjust against real transcripts.
- **Playbook-owned briefing shape.** Which tiers a pack wants, and in what
  voice, is ultimately playbook config — but that belongs with the pack SDK
  (`006`), not here. This doc ships one built-in briefing for the field-service
  pack.
